# 📋 작업 시작 가이드 (Global Instructions)

작업을 시작하기 전에 반드시 읽어주세요.

---

## 🚀 모든 작업 시작 전 필수 사항

### 1️⃣ **DISPATCH 폴더 구조 이해하기**

```
DISPATCH/
├── context/              👤 개인 정보 & 작업 규칙
├── projects/             ⚙️  현재 진행 중인 작업
│   ├── content/         📚 프로젝트 문맥
│   ├── data/            📊 입력 데이터
│   ├── work/            💻 작업 파일
│   └── scripts/         🐍 자동화 스크립트
├── outputs/             📤 최종 결과물 (최상위)
│   ├── reports/         📋 보고서
│   ├── plans/           📅 계획/플랜
│   └── logs/            📝 작업 로그
└── templates/           📑 템플릿 & 가이드
```

### 2️⃣ **작업 전 확인할 문서들** (우선순위 순)

```
1순위: context/working-rules.md
       → 현재 진행 중인 프로젝트의 작업 규칙 확인

2순위: CLAUDE.md (DISPATCH 폴더 최상위)
       → 전역 개발 원칙 확인
       → 기술 스택, 코딩 스타일 확인

3순위: projects/content/ 폴더
       → 현재 프로젝트의 문맥 자료 확인
       → 이전 작업 내용 파악

4순위: templates/global-instructions.md
       → 전체 작업 프로세스 확인
```

### 3️⃣ **불명확한 점이 있으면 반드시 질문하기**

❌ **하지 말 것**
- 추측으로 진행하기
- 불확실한 정보로 작업하기
- 파일 위치 임의로 정하기

✅ **할 것**
- 즉시 질문하기
- 명확한 답변 받고 진행
- 필요하면 기존 파일 확인

### 4️⃣ **실행 전 계획 제시 및 승인**

모든 작업 시작 전:
1. 작업 계획 간단히 정리
2. 저장할 파일 위치 명시
3. 사용자 승인 대기
4. 승인 후 실행

## 💬 커뮤니케이션 스타일

### 언어
- ✅ **한국어**로 모든 응답
- ✅ 기술 용어도 쉬운 한글로 설명
- ✅ 코드/파일명은 영문 유지

### 톤 (Tone)
```
전문적 + 친근함 = 최적 균형

예시:
❌ "시스템 아키텍처를 재구성하겠습니다." (너무 딱딱함)
✅ "구조를 재정리해서 더 깔끔하게 만들겠습니다." (친근함)

❌ "이거 그냥 해요 ㅋㅋ" (너무 편함)
✅ "이렇게 진행하겠습니다." (전문적 + 친근)
```

### 문장 길이
- 간결하고 핵심 위주
- 불필요한 서론/설명 생략
- 불릿 포인트 활용
- 3줄 이상 같은 주제는 번호 매기기

### 예시

**나쁜 예 (길고 불필요함)**
```
안녕하세요. 이번에는 배관설계 프로그램에 대해서
종합적으로 분석해보도록 하겠습니다. 우선 여러 가지
프로그램들이 있는데요, 각각의 특징과 가격, 그리고...
```

**좋은 예 (간결하고 핵심)**
```
배관설계 프로그램 비교 분석을 시작하겠습니다.

주요 내용:
1. 6개 프로그램 가격/기능 비교
2. 산업별 추천 솔루션
3. 도입 절차 및 비용
```

## 📁 결과물 저장 규칙

### 파일 저장 위치

| 결과물 종류 | 저장 위치 | 예시 |
|-----------|---------|------|
| **분석/조사** | `outputs/reports/` | 배관설계 프로그램 분석 |
| **계획/제안** | `outputs/plans/` | 프로젝트 계획서 |
| **코드** | `projects/work/` | Python 스크립트 |
| **데이터** | `projects/data/` | CSV, JSON 파일 |
| **로그** | `outputs/logs/` | 작업 로그 |
| **보고서** | `outputs/reports/` | 월간 보고서 |

### 파일 이름 규칙

**형식**: `YYYY-MM-DD_[주제]_V1`

**예시**:
```
✅ 좋은 예
- 2026-03-28_배관설계프로그램분석_V1.docx
- 2026-03-28_삼성평택공장현황_V1.xlsx
- 2026-03-25_SUS아르곤용접설명_V1.md

❌ 나쁜 예
- 분석.docx (날짜, 버전 없음)
- 배관프로그램_최종.xlsx (버전 불명확)
- 2026_03_28_파일명_FINAL.md (형식 불일치)
```

### 파일 형식 우선순위

1️⃣ **Word 문서** (.docx) - 보고서, 정식 문서
2️⃣ **마크다운** (.md) - 설명, 가이드, 기술 문서
3️⃣ **Excel** (.xlsx) - 데이터, 비교표, 분석
4️⃣ **PDF** - 최종 배포용

### 품질 확인 체크리스트

저장 전 확인:
- ✅ 파일명이 규칙을 따르는가?
- ✅ 저장 위치가 맞는가?
- ✅ 내용이 완전한가?
- ✅ 오류나 누락이 없는가?
- ✅ 형식이 일관성 있는가?

## ❓ 확실하지 않을 때

### 판단 기준

**❌ 멈추고 질문해야 할 때**
- 작업 범위가 불명확할 때
- 파일 저장 위치가 헷갈릴 때
- 여러 선택지가 있을 때
- 의존성이 있는 작업일 때
- 중요한 결정이 필요할 때

**✅ 진행해도 될 때**
- 이전 작업과 동일한 방식일 때
- 명확한 지침이 있을 때
- 영향이 작은 작업일 때

### 질문하는 방법

```
예: "마크다운 형식으로 저장할까요, 아니면 Word로 할까요?"

이렇게 선택지를 제시하면서 물어보기
→ 사용자가 빠르게 선택 가능
```

```

   ↓

   ↓
6. 사용자 승인 대기
   ↓
7. 승인 후 작업 진행
   ↓
8. 파일명 규칙에 맞게 저장
   ↓
9. 저장 위치와 내용 요약 보고
```
